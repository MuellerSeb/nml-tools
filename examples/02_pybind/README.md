# nml-tools pybind example

This example builds a small Python package with a package-local f2py extension. A
static Fortran library owns the generated namelist type and a persistent
`type(nml_config_t), target, save` instance. The f2py extension is only an
adapter: it links against that library and configures the persistent instance
through an opaque integer handle.

The split mirrors the intended use in larger projects:

- `fortran/generated/nml_helper.f90`, `fortran/generated/nml_config.f90`, and
  `fortran/config_store.f90` are compiled into the `nml_pybind_config` static
  library.
- `f2py/generated/f2py_config.f90` is generated by `nml-tools` and wraps the
  generated namelist type-bound procedures.
- `f2py/f2py_config_store.f90` is maintainer-owned. It exposes the library's
  handle/getter/reset routines to Python.

## Regenerate bindings

The generated files are checked in so the package can be built directly. To regenerate
them from the schema and `nml-config.toml`, run from the repository root:

```bash
PYTHONPATH=src python examples/02_pybind/scripts/generate.py
```

## Build and test

From this directory:

```bash
python -m pip install -v .[test]
python -m pytest tests
```

## Usage

```python
from nml_pybind_example import get_config, get_iterations, get_weights

cfg = get_config()
cfg.set(iterations=20, tolerance=1.0e-4, weights=1.0)
cfg.is_valid()

print(get_iterations())
print(get_weights())
```

Scalar array values are normalized by the generated Python shim before calling f2py.
For `weights=1.0`, only the first array entry is overwritten, matching the generated
Fortran `set` method's partial-array behavior; the remaining entries keep their
defaults.

The f2py-facing Fortran wrappers deliberately avoid `optional` dummy arguments
and assumed-shape arrays. The Python shim passes `has_<name>` flags plus dummy
values for omitted optional fields. The Fortran wrapper converts those flags to
allocated or unallocated local allocatables before calling the generated
type-bound `set` method, so omitted values still behave like absent optional
arguments in normal Fortran.
